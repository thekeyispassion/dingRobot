# 会议室管理 — 操作手册

## 数据来源

与 `meeting-room` Skill 共用钉钉 AI 表格 **「AI会议室预约助手」** 中的 3 张表：

| 表名 | 本模块用途 |
|------|-----------|
| 会议室信息 | 增删改查会议室（名称、楼栋、楼层、容量、设备、状态、备注） |
| 预约记录 | 停用/改设备前检查受影响用户 |
| 管理员 | 敏感操作权限校验——规则见 `manage-admin/SKILL.md` |

> 完整表结构见 `deploy-ai-table/SKILL.md`。管理员账号的添加/删除由 `manage-admin` Skill 负责。

## 权限模型

按动作敏感度分三级，不是一刀切：

| 级别 | 控制方式 | 适用操作 |
|------|---------|---------|
| **开放** | 任何人可操作 | 列出会议室、启用会议室、修改备注 |
| **协商确认** | 查受影响用户 → 通知 → 收集回复 → 执行 | 停用会议室、修改设备 |
| **管理员** | 双重匹配校验（见 `manage-admin/SKILL.md`） | 添加会议室、修改名称/楼栋/楼层/容量、删除会议室 |

**设计原则：** 设备坏了这种事，任何发现问题的用户都应该能触发报修流程。Agent 的角色不是"开门/关门"，而是信息中转——通知受影响的人，让他们自己选择。

## 操作方式

Agent 通过 MCP 工具直接读写钉钉 AI 表格。执行前用 ToolSearch 搜索相关 MCP 工具（关键词: `dingtalk`、`ai_table`、`record`）。

## 触发规则

| 场景 | 关键词/意图 | 执行操作 |
|------|------------|---------|
| 查看所有会议室 | 全部会议室、列出房间、房间列表、有哪些房间 | 操作1: 列出所有会议室 |
| 添加会议室 | 添加、新增、加一间、增加 | 操作2: 添加会议室 |
| 修改会议室 | 修改、改一下、更新、调整 | 操作3: 修改会议室 |
| 停用会议室 | 停用、禁用、关闭、暂停使用、坏了、报修 | 操作4: 停用会议室 |
| 启用会议室 | 启用、恢复、重新开放 | 操作5: 启用会议室 |

> 用户说"修改某某房间"但没有说改什么时，先列出该房间当前信息，再追问具体要改什么。

## 可用操作

### 操作1: 列出所有会议室

**权限：开放**（任何人可查看）

**步骤：**
1. 查「会议室信息」表
   - 默认全部列出（含维护中的）
   - 筛选项 `状态 = '可用'` → 仅可用
2. 按 `楼栋`、`楼层`、`会议室名称` 排序

**返回：** `{"success": true, "rooms": [...], "count": N}`

---

### 操作2: 添加会议室

**权限：管理员**（双重匹配校验，规则见 `manage-admin/SKILL.md`）

**步骤：**
1. **权限校验：** 双重匹配：
   - 优先查 `用户ID = 当前userId` → 命中 → 管理员
   - 降级查 `用户ID = "" AND 用户姓名 = 当前userName` → 命中 → 管理员（降级）
   - 都不匹配 → 拒绝（规则详见 `manage-admin/SKILL.md`）
2. **重名检查：** 查「会议室信息」表，筛选 `会议室名称 = 新名称` → 已存在则拒绝
3. **新增记录：** 在「会议室信息」表新增一行：
   - 会议室名称、楼栋、楼层（整数）、容量（整数）、设备、备注
   - 状态 = '可用'（默认）

**参数：** USER_ID、名称、楼栋、楼层(整数)、容量(整数)、设备、备注

**返回：** `{"success": true, "message": "会议室 'xxx' 添加成功", "room": {...}}`

**错误情况：**
- 非管理员 → 权限不足
- 重名 → "会议室 'xxx' 已存在"

---

### 操作3: 修改会议室

**权限：按修改字段分级**

| 字段 | 权限级别 | 控制方式 |
|------|---------|---------|
| 备注 | 开放 | 任何人可改 |
| 设备 | 协商确认 | 通知今天有预约的用户 |
| 会议室名称、楼栋、楼层、容量 | 管理员 | 双重匹配校验 |

**步骤：**

1. **查房间：** 查「会议室信息」表按记录 ID 找房间 → 不存在则拒绝
2. **权限校验（按字段）：**
   - 只改备注 → 跳过校验，直接更新
   - 包含设备 → 走协商流程（见下方）
   - 包含名称/楼栋/楼层/容量 → 双重匹配校验（规则见 `manage-admin/SKILL.md`）
3. **改名查重：** 如果传了名称且与当前不同，查「会议室信息」表确认未被占用
4. **更新记录：** 更新指定字段
5. **不可修改状态字段——停用/启用用操作4/5**

**设备修改协商流程：**
```
改设备（如"投影仪坏了"）→ 
  查「预约记录」表：会议室=目标房间, 日期=今天, 状态=已确认 →
  有预约 → 通知受影响用户："xx会议室的设备有变更，你的预约（xx时间）不受影响"
  无预约 → 直接更新
```

> 设备字段的协商是**通知**而非**征求同意**——改设备不影响预约本身，只是告知变化。

**参数：** USER_ID、房间记录 ID、要修改的字段+新值

**返回：** `{"success": true, "message": "会议室 ID:xxx 修改成功：设备 → 投影仪,白板（已通知今天预约用户）"}`

**错误情况：**
- 无有效字段 → 列出可修改字段
- 名称被占用 → 拒绝
- 管理员字段非管理员操作 → 权限不足

---

### 操作4: 停用会议室

**权限：协商确认**（任何人可触发，Agent 协调通知和确认）

这是最核心的流程变化。设备坏了不应该卡在"找管理员"——任何发现问题的用户都能触发停用流程。

**完整流程：**

**第1步：查询受影响预约**

查「预约记录」表，筛选：
- `会议室 = 目标房间`
- `日期 >= 今天`
- `状态 = '已确认'`

**第2步：分支处理**

```
┌─ 无预约 → 直接停用
│   更新「会议室信息」表：状态 = '维护中'
│   返回："302会议室已停用，当前无活跃预约。"
│
└─ 有预约 → 进入协商流程（第3步）
```

**第3步：通知受影响用户**

Agent 在群里列出受影响的人和时间：

> "302会议室需要停用。今天有 N 个预约：
> - @李四 14:00-16:00
> - @王五 16:00-18:00
> 需要我帮你通知他们吗？"

用户确认后，Agent 逐个通知：

> "@李四 302会议室需要停用，你的预约（14:00-16:00）需要取消还是保留？回复'取消'或'保留'。"

**第4步：收集回复 + 执行**

- 回复「取消」→ 在「预约记录」表中将该预约状态更新为 '已取消'
- 回复「保留」→ 预约保留不变
- 超时未回复 → 保留预约（不替别人做决定）

**第5步：完成停用 + 汇总**

更新「会议室信息」表：状态 = '维护中'

> "302会议室已设为'维修中'，新预约暂停。
> 处理结果：
> - 李四 14:00-16:00：已取消 ✓
> - 王五 16:00-18:00：保留（到时仍可使用）
> 王五的预约结束后请通知维修人员。"

**紧急情况：**

如果用户明确说"紧急"或"立刻停用"：

> "已将 302 设为'维修中'。预约保留，但会通知他们会议室状态变更。
> 如需强制取消所有预约，请说'强制取消 302 所有预约'——这个操作需要管理员权限。"

**参数：** USER_ID、房间记录 ID

**返回：** 流程中的每一步都有对应的用户可见输出，最终结果包含处理汇总。

---

### 操作5: 启用会议室

**权限：开放**（任何人可操作）

停用期间没有新预约产生，启用不影响任何人，无需管理员。

**步骤：**
1. **查房间：** 查「会议室信息」表按记录 ID 找房间 → 不存在则拒绝
2. **状态检查：** 如果 `状态 = '可用'` → 拒绝（已启用）
3. **执行启用：** 更新「会议室信息」表，设置 `状态 = '可用'`

**参数：** 房间记录 ID

**返回：** `{"success": true, "message": "会议室 '信电楼330' 已恢复可用"}`
