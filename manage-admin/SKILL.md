# 管理员账号管理 — 操作手册

## 概述

管理「管理员」表中的记录。核心难点：用户说"加张三"时只有姓名没有 userId。本 Skill 通过钉钉通讯录 MCP 自动解析。

## 依赖的 MCP

| MCP Server | 用途 |
|-----------|------|
| 钉钉通讯录 | 用户名 → userId 解析（`search_contact_by_key_word`） |
| 钉钉 AI 表格 | 读写「管理员」表 |

## 数据表

操作钉钉 AI 表格 **「AI会议室预约助手」** 中的「管理员」表：

| 字段 | 类型 | 说明 |
|------|------|------|
| 用户ID | text | 钉钉 userId，正常情况必填。降级场景可为空字符串 |
| 用户姓名 | text | 必填 |
| 角色 | text | 默认 admin |
| 创建时间 | text | 添加时间 |

|------|-------|---------|

## 可用操作

### 操作1: 添加管理员

**权限：管理员**（只有现有管理员才能添加新管理员）

**完整流程：**

```
Step 1: 权限校验
查「管理员」表校验当前用户是管理员（见下方「权限校验规则」）
→ 非管理员 → 拒绝

Step 2: 解析用户名 → userId
使用钉钉通讯录 MCP: search_contact_by_key_word(用户姓名)
   │
   ├─ 1 个结果 → userId = xxx, name = 通讯录中的姓名
   │
   ├─ 多个结果 → 列出让管理员选:
   │   进一步调 get_user_info_by_user_ids 获取部门信息
   │   "找到 3 个张三：
   │    1. 张三 — 信电学院
   │    2. 张三 — 理学院
   │    3. 张三 — 计算机学院
   │    你要添加哪个？"
   │   管理员选一个 → userId = 所选
   │
   └─ 0 个结果 → 降级处理（见下方）

Step 3: 去重检查
查「管理员」表，筛选:
  - 用户ID = userId（有 userId 时）
  - 用户ID = "" AND 用户姓名 = 目标姓名（降级时）
→ 已有记录 → "xxx 已经是管理员"

Step 4: 写入
在「管理员」表新增一行:
  用户ID = userId（降级则为 ""）
  用户姓名 = 通讯录返回的正式姓名
  角色 = admin
  创建时间 = 当前时间
```

**降级处理（搜不到 userId）：**

```
通讯录搜不到 → 无法确认用户身份

Agent 提示:
  "通讯录中未找到'张三'的userId，可通过姓名匹配添加。
   风险：任何叫'张三'的人都会被识别为管理员。
   是否继续？"

管理员确认 →
  写入: 用户ID = ""（空字符串标记位）, 用户姓名 = "张三", 角色 = admin

管理员拒绝 →
  停止操作:
  "已取消。如需添加，请确保'张三'在通讯录中，或提供准确的钉钉账号。"
```

**参数：** USER_ID（操作者）、目标用户姓名

**返回示例：**

正常:
```
✅ 已将「张三」添加为管理员
  userId: 03336453153221449286
  角色: admin
```

降级:
```
⚠️ 已将「张三」添加为管理员（仅姓名匹配）
  注：通讯录未找到该用户，权限校验时将仅通过姓名识别。
  如有同名用户，请手动补全 userId。
```

### 操作2: 删除管理员

**权限：管理员**（只有现有管理员才能删除）

**步骤：**

1. **权限校验：** 查「管理员」表确认当前用户是管理员
2. **查找目标：** 查「管理员」表，筛选 `用户姓名 = 目标姓名`
   - 有多条 → 列出让操作者选
   - 0 条 → "未找到管理员'张三'"
3. **确认删除：** 列出要删除的记录并确认
4. **执行：** 删除该记录

**约束：** 不能删除最后一个管理员——至少保留一个管理员防止锁死。

**参数：** USER_ID（操作者）、目标用户姓名

**返回：** `"✅ 已将'张三'从管理员移除"`

---

### 操作3: 查看管理员列表

**权限：开放**（任何人可查看）

**步骤：** 查「管理员」表全部记录

**返回格式：**
```
📋 当前管理员（N 人）：
  1. 张三 — admin（userId: 033364...）
  2. 李四 — admin（userId: 022365...）⚠️ 降级匹配（仅姓名）
```

降级匹配的管理员标注 ⚠️，提醒可能需要补全 userId。

---

### 操作4: 补全 userId

**权限：管理员**（只有现有管理员才能补全）

**场景说明：** 管理员在钉钉 AI 表格中手动添加了新行（只填了姓名，userId 留空），然后让 Agent 补全。Agent 必须通过通讯录查出真实 userId 并写入，**绝对禁止使用操作者自己的 userId**。

**完整流程：**

```
Step 1: 权限校验
查「管理员」表校验当前用户是管理员 → 非管理员 → 拒绝

Step 2: 查询目标
查「管理员」表，筛选 用户ID = "" 的所有行
  ├─ 0 条 → "所有管理员的 userId 已完整，无需补全"
  └─ N 条 → 列出需要补全的管理员:
       "以下管理员需要补全 userId：
        1. 张三（记录ID: rec_xxx）
        2. 李四（记录ID: rec_yyy）
        要补全哪个？回复数字序号"

Step 3: 去通讯录查真实 userId（🚨 关键：查的是目标人，不是操作者自己）
对目标姓名，调 search_contact_by_key_word(目标用户姓名)
   ├─ 1 个结果 → userId = 通讯录返回的真实userId
   ├─ 多个结果 → 列出让管理员选
   └─ 0 个结果 → 保持空，提示无法补全

Step 4: 写入目标记录
更新「管理员」表中匹配的记录: 用户ID = 通讯录返回的真实userId
```

**🚨 绝对禁止行为：**

| ❌ 错误做法 | ✅ 正确做法 |
|------------|------------|
| 查不到 → 填操作者自己的 userId | 查不到就保持空 |
| 懒得搜直接填操作者 userId | 必须去通讯录搜 |
| 搜索结果里有操作者就填进去 | 同名人让用户选，选错比留空更糟 |
| 误以为"补管理员id"是补自己的id | 补的是用户ID为空的那个人 |

**典型对话示例：**

```
管理员: "补全管理员id"

Agent: 查「管理员」表 → 发现 1 条用户ID为空:
  "以下管理员需要补全 userId：
   1. 张三（记录ID: rec_abc123）
   要补全张三的 userId 吗？"

管理员: "补全"

Agent: search_contact_by_key_word("张三")
  → 1 个结果: userId=022365xxx
  → 更新 rec_abc123: 用户ID = "022365xxx"

Agent: "✅ 已补全张三的 userId: 022365xxx"
      如果查不到: "通讯录中未找到'张三'，暂时无法补全。"
      如果已完整: "所有管理员的 userId 已完整，无需补全。"
```

## 权限校验规则（所有 Skill 共用）

管理员校验逻辑——不仅仅在本 Skill 中使用，`room_manager` 和 `meeting_room` 的敏感操作也都用这个规则：

```
查「管理员」表:
  
  第一优先：userId 精确匹配
    筛选 用户ID = 当前用户userId → 有记录 → ✅ 管理员
  
  第二优先：降级姓名匹配
    筛选 用户ID = "" AND 用户姓名 = 当前用户userName → 有记录 → ⚠️ 管理员（降级）
  
  都不匹配 → ❌ 非管理员

提示: 如果命中降级匹配，在正常操作中不显示警告（用户无感知），
     但在「查看管理员列表」时标注 ⚠️ 提示管理员可能需要补全 userId。
```

**为什么这个规则放在 manage-admin 而不是 room_manager？**

因为管理员数据的增删改都在这里，校验规则跟着数据走——改一个地方就行。

## 两条业务线统一

| 场景 | 流程 | userId 来源 |
|------|------|------------|
| 对话添加 | "把张三加为管理员" → Agent 调通讯录 MCP → 写表 | MCP 自动解析 |
| 表格编辑 | 管理员手动在表格加行（只填了姓名） → 让 Agent 补全 | 执行操作4「补全 userId」 |

> 不再被动补全——改为管理员主动触发操作4。避免 Agent 在权限校验时偷偷改数据。
