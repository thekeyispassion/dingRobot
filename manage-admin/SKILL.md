# 管理员账号管理 — 操作手册

## 概述

管理「管理员」表中的记录。核心难点：用户说"加张三"时只有姓名没有 userId。本 Skill 通过钉钉通讯录 MCP 自动解析。

## 依赖的 MCP

| MCP Server | 用途 |
|-----------|------|
| 钉钉通讯录 | 用户名 → userId 解析（`search_contact_by_key_word`） |
| 钉钉 AI 表格 | 读写「管理员」表 |

## 数据表

操作钉钉 AI 表格 **「AI会议室预约助手」** 中的「管理员」表：

| 字段 | 类型 | 说明 |
|------|------|------|
| 用户ID | text | 钉钉 userId，正常情况必填。降级场景可为空字符串 |
| 用户姓名 | text | 必填 |
| 角色 | text | 默认 admin |
| 创建时间 | text | 添加时间 |

## 可用操作

### 操作1: 添加管理员

**权限：管理员**（只有现有管理员才能添加新管理员）

**完整流程：**

```
Step 1: 权限校验
查「管理员」表校验当前用户是管理员（见下方「权限校验规则」）
→ 非管理员 → 拒绝

Step 2: 解析用户名 → userId
使用钉钉通讯录 MCP: search_contact_by_key_word(用户姓名)
   │
   ├─ 1 个结果 → userId = xxx, name = 通讯录中的姓名
   │
   ├─ 多个结果 → 列出让管理员选:
   │   进一步调 get_user_info_by_user_ids 获取部门信息
   │   "找到 3 个张三：
   │    1. 张三 — 信电学院
   │    2. 张三 — 理学院
   │    3. 张三 — 计算机学院
   │    你要添加哪个？"
   │   管理员选一个 → userId = 所选
   │
   └─ 0 个结果 → 降级处理（见下方）

Step 3: 去重检查
查「管理员」表，筛选:
  - 用户ID = userId（有 userId 时）
  - 用户ID = "" AND 用户姓名 = 目标姓名（降级时）
→ 已有记录 → "xxx 已经是管理员"

Step 4: 写入
在「管理员」表新增一行:
  用户ID = userId（降级则为 ""）
  用户姓名 = 通讯录返回的正式姓名
  角色 = admin
  创建时间 = 当前时间
```

**降级处理（搜不到 userId）：**

```
通讯录搜不到 → 无法确认用户身份

Agent 提示:
  "通讯录中未找到'张三'的userId，可通过姓名匹配添加。
   风险：任何叫'张三'的人都会被识别为管理员。
   是否继续？"

管理员确认 →
  写入: 用户ID = ""（空字符串标记位）, 用户姓名 = "张三", 角色 = admin

管理员拒绝 →
  停止操作:
  "已取消。如需添加，请确保'张三'在通讯录中，或提供准确的钉钉账号。"
```

**参数：** USER_ID（操作者）、目标用户姓名

**返回示例：**

正常:
```
✅ 已将「张三」添加为管理员
  userId: 03336453153221449286
  角色: admin
```

降级:
```
⚠️ 已将「张三」添加为管理员（仅姓名匹配）
  注：通讯录未找到该用户，权限校验时将仅通过姓名识别。
  如有同名用户，请手动补全 userId。
```

### 操作2: 删除管理员

**权限：管理员**（只有现有管理员才能删除）

**步骤：**

1. **权限校验：** 查「管理员」表确认当前用户是管理员
2. **查找目标：** 查「管理员」表，筛选 `用户姓名 = 目标姓名`
   - 有多条 → 列出让操作者选
   - 0 条 → "未找到管理员'张三'"
3. **确认删除：** 列出要删除的记录并确认
4. **执行：** 删除该记录

**约束：** 不能删除最后一个管理员——至少保留一个管理员防止锁死。

**参数：** USER_ID（操作者）、目标用户姓名

**返回：** `"✅ 已将'张三'从管理员移除"`

---

### 操作3: 查看管理员列表

**权限：开放**（任何人可查看）

**步骤：** 查「管理员」表全部记录

**返回格式：**
```
📋 当前管理员（N 人）：
  1. 张三 — admin（userId: 033364...）
  2. 李四 — admin（userId: 022365...）⚠️ 降级匹配（仅姓名）
```

降级匹配的管理员标注 ⚠️，提醒可能需要补全 userId。

---

### 操作4: 补全 userId

**权限：管理员**（只有现有管理员才能补全）

**场景说明：** 管理员在钉钉 AI 表格中手动添加了新行（只填了姓名，userId 留空），然后让 Agent 补全。Agent 全自动批量处理所有空 userId 记录，**无需用户逐个选择**。**绝对禁止使用操作者自己的 userId**。

**完整流程：**

```
Step 1: 权限校验
查「管理员」表校验当前用户是管理员 → 非管理员 → 拒绝

Step 2: 批量查询所有待补全记录
查「管理员」表，筛选 用户ID = "" 的所有行
  ├─ 0 条 → "所有管理员的 userId 已完整，无需补全" → 结束
  └─ N 条 → 进入 Step 3，逐条自动处理

Step 3: 逐条自动补全（无需用户参与）
对每一条 用户ID="" 的记录:
  调 search_contact_by_key_word(用户姓名)
     ├─ 1 个结果 → 直接更新 用户ID = 真实userId ✅
     ├─ 多个结果 → 跳过（同名无法自动区分，保留空）
     └─ 0 个结果 → 跳过（保留空）

Step 4: 汇总结果
全部处理完后，一次性输出:
  ✅ 补全 2 人: 张三 , 李四 
  ⚠️ 跳过 1 人: 王五（通讯录多名匹配，无法自动确定）
```

**关键规则：**

| 规则 | 说明 |
|------|------|
| 全自动 | 不逐条询问用户，查到就写，查不到就跳 |
| 包括操作者自己 | 如果管理员自己的 userId 也是空的，一并补全 |
| 同名多匹配 → 跳过 | 无法自动确定，保留空等用户手动处理 |
| 搜不到 → 跳过 | 保留空 |
| 禁止填操作者id | 每个目标的 userId 必须来自通讯录搜索该目标姓名 |

**典型对话示例：**

```
管理员: "补全管理员id"

Agent: 查「管理员」表 → 发现 3 条用户ID为空
  → 逐条自动搜索通讯录...
  → 张三: 1人匹配 → 已补全 
  → 李四: 1人匹配 → 已补全 
  → 王五: 多人匹配 → 跳过

Agent: "✅ 补全 2 人: 张三 , 李四 
        ⚠️ 跳过 1 人: 王五（通讯录多名匹配，无法自动确定）"
```

```
管理员: "补全管理员id"

Agent: 查「管理员」表 → 所有 userId 已完整
Agent: "所有管理员的 userId 已完整，无需补全。"
```

## 权限校验规则（所有 Skill 共用）

管理员校验逻辑——不仅仅在本 Skill 中使用，`room_manager` 和 `meeting_room` 的敏感操作也都用这个规则：

```
查「管理员」表:
  
  第一优先：userId 精确匹配
    筛选 用户ID = 当前用户userId → 有记录 → ✅ 管理员
  
  第二优先：降级姓名匹配
    筛选 用户ID = "" AND 用户姓名 = 当前用户userName → 有记录 → ⚠️ 管理员（降级）
    → 命中后自动尝试补全：调通讯录 MCP 搜索该用户姓名 → 找到则更新 userId
    → 补全成功 → 下次走精确匹配；补全失败 → 仍按降级通过，不阻塞操作
  
  都不匹配 → ❌ 非管理员

提示: 自动补全在后台静默执行，不打扰用户。操作完成后不显示补全结果——
     仅当用户主动查「管理员列表」时标注 ⚠️ 降级匹配。
```

**为什么这个规则放在 manage-admin 而不是 room_manager？**

因为管理员数据的增删改都在这里，校验规则跟着数据走——改一个地方就行。

## 两条业务线统一

| 场景 | 流程 | userId 来源 |
|------|------|------------|
| 对话添加 | "把张三加为管理员" → Agent 调通讯录 MCP → 写表 | MCP 自动解析 |
| 表格编辑 | 管理员手动在表格加行（只填了姓名） → 让 Agent 补全 | 执行操作4「补全 userId」 |

> 不再被动补全——改为管理员主动触发操作4。避免 Agent 在权限校验时偷偷改数据。
