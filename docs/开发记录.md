**项目：** AI 会议室预约助手（基于 OpenClaw + 钉钉群机器人）  
**开发者：** DDTalk Dev  
**开发周期：** 2026-07-13  
**开发方式：** Vibe Coding（AI 驱动的对话式编程）  

---

## 一、项目背景

用一句话描述：**"一句话，约一间房。"**

学院会议室预约流程繁琐（登录系统、选楼栋、填信息，10+ 次点击），师生体验差。目标是开发一个钉钉群机器人，@一下说句话就能完成预约。

技术栈：OpenClaw（小龙虾 Agent）+ Python Skills + SQLite + 钉钉机器人 API + 国产大模型。

### 阶段 2：核心实现（约 40 分钟）

采用了 **Subagent-Driven Development** 模式：每个 Task 派发给独立的 AI subagent，写完一个 review 一个。

#### Task 1-2：基础设施
| Task | 内容 | 耗时 | 结果 |
|------|------|------|------|
| 1 | 项目骨架 + SQL Schema | ~3 分钟 | 5 文件，2 表 3 索引 |
| 2 | 数据库管理器 | ~1 分钟 | db_manager.py 核心模块 |

**踩坑：** 服务器没装 `sqlite3` CLI，验证 SQL 时用 Python 的 `sqlite3` 模块代替——AI subagent 自动适配了。

#### Task 3-6：业务逻辑
| Task | 内容 | 实现细节 | 测试 |
|------|------|---------|------|
| 3 | 模糊时间解析 | "明天下午"→标准时段，12 种表达 | 13 pass |
| 4 | 房间查询 | 空闲查询、预约总览、模糊匹配 | 7 pass |
| 5 | 预约核心 | 冲突检测、替代推荐、相邻时段不重叠 | 8 pass |
| 6 | 预约管理 | 取消+权限隔离、我的预约 | 6 pass |

**关键决策：**
- 冲突检测用 `<` `>`（不含等于），相邻时段不互斥（14:00-16:00 后面可以约 16:00-18:00）
- 推荐算法：同楼栋优先 > 容量差异升序，Top 3
- 每个 Skill 函数输入/输出都是 JSON 字符串，对接 OpenClaw

#### Task 7-9：提示词 + CLI + 测试
| Task | 内容 | 产出 |
|------|------|------|
| 7 | LLM 提示词 | 系统提示词 + 5 种意图分类 + 12 种回复模板 |
| 8 | CLI 测试入口 | 300 行交互式命令行，模拟钉钉对话 |
| 9 | 集成测试 | 19 个端到端用例 |

## 五、Vibe Coding 心得

### 做得好的

1. **先设计再编码** — 和 AI 花 20 分钟对齐设计，比直接写代码节省了后续反复修改的时间
2. **Subagent-Driven** — 每个 Task 独立派发、独立 review，质量明显比一口气写完高
3. **TDD 写得很实** — 53 个测试用例都是真的在验证业务逻辑，不是充数的
4. **Review 环节有价值** — AI reviewer 发现了几个代码实现和 spec 的细微偏差

### 可以更好的

1. **计划文档的代码模板有 bug** — time_parser 的 reference code 缺少对无效输入的校验，AI subagent 自己发现并修正了
2. **任务粒度还可以更细** — Task 8（CLI）300 行代码一个 Task 就写完了，review 时跳过了很多细节
3. **并行度不够** — 像 Task 3（time_parser）和 Task 7（prompts）完全独立，理论上可以并行，但串行执行了

### 如果再来一次

- 会在设计文档里就把数据格式约定（JSON schema）写得更精确
- 会把 prompts 文件和 skills 代码放一起 Task 做（提示词和对应的 skill 是一对一的）
- 会给 CLI 也写单元测试

## 六、Day 2：接口层补充（2026-07-14）

### 背景

Day 1 完成了 Skills 业务逻辑层，但有两个关键问题：

1. **意图分类是硬编码关键词** — 在 `cli/test_shell.py` 里用 if-else 匹配，没有真正调 LLM
2. **钉钉接口缺失** — 用户身份是 Mock 的，消息格式没有对接真实钉钉
3. **API Key 没地方放** — 如果硬编码在代码里，git push 会泄露

Day 2 的目标：**把接口层从 CLI 中剥离，独立成模块，支持 LLM API 调用 + 钉钉消息处理 + 隐私安全。**

### 做了什么

新增 `interfaces/` 层，3 个模块：

| 模块 | 职责 | 关键设计 |
|------|------|---------|
| `config.py` | 配置管理 | config.yaml → 环境变量 → 默认值，三层优先级 |
| `llm_client.py` | LLM 客户端 | OpenAI 兼容协议，无 API Key 时自动降级到本地关键词匹配 |
| `dingtalk_handler.py` | 钉钉消息 | 兼容 OpenClaw 格式 + 钉钉原始 webhook 格式 |

**隐私安全措施：**
- `config.yaml` → `.gitignore` 保护，永不提交
- `config.example.yaml` → 只有占位符，可以安全提交
- 支持环境变量 `DDTALK_LLM_KEY` 等作为备选
- `mask_key()` 函数脱敏显示密钥（日志/调试用）
- 代码中零硬编码密钥

**CLI 重构：**
- 删除了 `cli/test_shell.py` 中约 70 行硬编码的 `classify_intent()` 和 `MockUser` 类
- 改为从 `interfaces` 导入，接口层可替换
- 新增 `config` 命令查看当前配置状态

### 踩坑

- **正则提取房间号** — `_extract_room_name_from_text("信电楼330")` 的正则 `[A-Za-z]*\d{3,4}` 只能提取到 `"330"`，不是全名 `"信电楼330"`。不过这不影响功能，因为 room_query 的 `get_room_by_name` 支持 LIKE 模糊匹配
- **测试写了 91 个** — Day 1 的 53 个 + Day 2 的 38 个，一次性全部通过

### Day 2 心得

1. **接口层分离后职责清晰很多** — CLI 只管交互循环，意图分类归 llm_client，消息解析归 dingtalk_handler，改哪个都不影响别的
2. **降级策略很重要** — llm_client 在没配 API Key 时自动回退到本地关键词，这样开发机上不用配 Key 也能跑，上了生产配 Key 就自动走 LLM
3. **隐私安全要在一开始就考虑** — 如果 Day 1 就把 API Key 写在代码里，git push 出去就收不回了。config.yaml 的 gitignore + example 模板是标准做法

### Day 2 收尾：一键验证脚本

写了一个 `verify.py`，4 层 23 项检查：

| 层次 | 内容 | 结果 |
|------|------|------|
| L1 | 数据库初始化 + 种子数据验证 | 6/6 ✅ |
| L2 | 全量 pytest（91 tests） | 9/9 模块 ✅ |
| L3 | CLI 功能测试（启停、预约、查询、总览、配置） | 8/8 ✅ |
| L4 | LLM 配置状态检查 | 信息展示 |

**踩坑：** CLI 子进程测试时，上一个测试创建的 `db/meeting_rooms.db` 会影响下一个测试（预约冲突），加了 `_reset_db()` 在每个测试前重建数据库解决。另外 PRAGMA table_info 在某些 Python 版本下返回格式不一致，改为直接查数据验证列存在。

### Day 2 补充：SKILL.md — OpenClaw 技能注册

**搞清楚了项目在 OpenClaw 里的正确位置：**

之前混淆了两个概念：
1. `skills/` 目录里的 Python 文件 —— 是**函数库**，不是 OpenClaw 的"Skill"
2. OpenClaw 的"Skill" —— 需要一个 `SKILL.md` 说明书，告诉 AI 有哪些能力、怎么调用

**SKILL.md 的作用：**
- 是 OpenClaw AI 的"使用手册"——AI 读取它来理解自己能做什么
- 包含 6 种操作的精确 bash 命令（查询空闲、预约总览、预约、我的预约、取消、时间解析）
- 包含学院 8 间会议室的速查表
- 定义了回复风格和注意事项

**架构理解的变化：**
```
之前（错误）:  用户 → LLM意图分类 → handle_command → skills
               ↑ 多了一层不必要的 LLM 调用

现在（正确）:  用户 → OpenClaw AI 读 SKILL.md → 直接 exec Python
               ↑ AI 本身就是最好的意图理解器
```

`interfaces/llm_client.py` 的意图分类只在 CLI 独立模式下作为降级方案使用，在 OpenClaw 环境下不需要——AI 自己就能理解"帮我约明天下午330"是什么意思。

1. `cp config.example.yaml config.yaml` 并填入真实 API Key

4. 配置 OpenClaw，将 `skills/` 目录注册为 Skill 目录
5. 在 OpenClaw 中设置 LLM 系统提示词为 `prompts/system_prompt.md`
6. 按 `技术说明.md` 完成钉钉机器人接入
